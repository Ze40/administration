# Отчет по приложению и Docker

## 1. Назначение проекта

В каталоге `docker` находится небольшое веб-приложение на Python, реализующее CRUD-операции для таблицы `users` в базе данных PostgreSQL.

Приложение предназначено для:

- создания пользователей;
- получения списка пользователей;
- получения пользователя по идентификатору;
- обновления данных пользователя;
- удаления пользователя.

Сервис реализован как HTTP API и запускается через `FastAPI`.

## 2. Используемые технологии

Зависимости проекта описаны в файле `pyproject.toml`.

Основной стек:

- `FastAPI` — веб-фреймворк для реализации API;
- `Uvicorn` — ASGI-сервер для запуска приложения;
- `SQLAlchemy` — ORM и инструмент для работы с базой данных;
- `psycopg` — драйвер PostgreSQL;
- `pytest` — библиотека для тестирования;
- `pytest-asyncio` — поддержка асинхронных тестов;
- `httpx` — HTTP-клиент, используемый в тестах.

## 3. Структура проекта

Основные файлы:

- `src/main.py` — точки входа API и обработчики HTTP-запросов;
- `src/database.py` — конфигурация подключения к базе данных;
- `src/models.py` — ORM-модель таблицы `users`;
- `src/schemas.py` — Pydantic-схемы запросов и ответов;
- `tests/conftest.py` — фикстуры для тестов;
- `tests/test_users.py` — тесты CRUD-функциональности;
- `Dockerfile` — инструкция сборки Docker-образа;
- `compose.yaml` — запуск приложения и PostgreSQL через Docker Compose.

## 4. Архитектура приложения

Приложение построено по простой классической схеме:

1. Клиент отправляет HTTP-запрос.
2. FastAPI принимает запрос и передает его соответствующему обработчику.
3. Обработчик работает с базой данных через SQLAlchemy.
4. Результат возвращается клиенту в формате JSON.

Программа использует асинхронную работу с базой данных через `AsyncSession`.

## 5. Работа с базой данных

Логика подключения к PostgreSQL находится в `src/database.py`.

Ключевые особенности:

- строка подключения берется из переменной окружения `DATABASE_URL`;
- если переменная не задана, используется значение по умолчанию:
  `postgresql+psycopg://kubsu:kubsu@127.0.0.1:5432/kubsu`;
- создается `engine` SQLAlchemy;
- создается фабрика асинхронных сессий `AsyncSessionLocal`;
- создается базовый класс `Base` для ORM-моделей.

Также реализованы:

- `init_db()` — создает таблицы при запуске приложения;
- `get_db()` — предоставляет сессию базы данных обработчикам запросов.

Такое решение делает приложение переносимым: оно может запускаться как локально, так и в контейнере, где адрес базы данных будет отличаться.

## 6. Модель данных

В `src/models.py` описана модель `User`.

Таблица `users` содержит:

- `id` — первичный ключ;
- `name` — имя пользователя.

Это минимальная модель, достаточная для демонстрации CRUD-операций.

## 7. Схемы API

В `src/schemas.py` описаны Pydantic-схемы:

- `UserCreate` — схема входных данных при создании и обновлении пользователя;
- `User` — схема ответа API.

Схема `User` включает:

- `id`;
- `name`.

Параметр `from_attributes = True` позволяет корректно преобразовывать ORM-объекты SQLAlchemy в JSON-ответы.

## 8. Реализованные HTTP-методы

Маршруты находятся в `src/main.py`.

### `POST /users/`

Создает нового пользователя.

Пример запроса:

```json
{
  "name": "John Doe"
}
```

Пример ответа:

```json
{
  "id": 1,
  "name": "John Doe"
}
```

### `GET /users/`

Возвращает список пользователей.

Поддерживает параметры:

- `skip` — смещение;
- `limit` — ограничение количества записей.

### `GET /users/{user_id}`

Возвращает пользователя по идентификатору.

Если пользователь не найден, приложение возвращает ошибку `404`.

### `PATCH /users/{user_id}`

Изменяет имя существующего пользователя.

Если пользователь не найден, возвращается `404`.

### `DELETE /users/{user_id}`

Удаляет пользователя.

Пример ответа:

```json
{
  "detail": "User deleted"
}
```

## 9. Инициализация приложения

При старте приложения вызывается функция `startup()` из `src/main.py`.

Она запускает `init_db()`, которая создает таблицы в базе данных.

Это позволяет запускать сервис без ручного предварительного создания структуры БД.

## 10. Тестирование

Тесты находятся в каталоге `tests`.

Проверяются следующие сценарии:

- создание пользователя;
- получение списка пользователей;
- получение одного пользователя;
- обновление пользователя;
- удаление пользователя;
- получение ошибки `404` для несуществующего пользователя.

В `tests/conftest.py` описаны фикстуры:

- инициализация базы данных;
- создание тестового клиента;
- очистка таблицы `users` перед тестами;
- создание тестового пользователя.

Тестирование позволяет убедиться, что основной CRUD-функционал работает корректно.

## 11. Dockerfile

Сборка контейнера описана в `Dockerfile`.

Используется двухстадийная сборка (`2-stage build`).

### Первый этап — `builder`

На этом этапе:

- берется образ `python:3.12-slim`;
- создается виртуальное окружение;
- копируются исходники и тесты;
- устанавливаются зависимости проекта.

Цель первого этапа — подготовить окружение приложения.

### Второй этап — `runtime`

На этом этапе:

- снова используется легковесный образ `python:3.12-slim`;
- из первого этапа копируется готовое виртуальное окружение;
- копируются только нужные файлы проекта;
- запускается `uvicorn`.

Команда запуска:

```sh
uvicorn src.main:app --host 0.0.0.0 --port ${PORT}
```

## 12. Почему выбран 2-stage build

Двухстадийная сборка нужна по нескольким причинам:

- уменьшает размер итогового образа;
- отделяет этап установки зависимостей от этапа запуска;
- делает образ чище;
- соответствует требованиям задания.

Такой подход считается хорошей практикой при контейнеризации приложений.

## 13. Docker context и .dockerignore

Файл `.dockerignore` исключает лишние файлы из контекста сборки.

Исключаются:

- `.git`;
- `.github`;
- `__pycache__`;
- `.pytest_cache`;
- `.venv`;
- временные Python-файлы;
- `.env`.

Это уменьшает размер контекста и ускоряет сборку образа.

## 14. Docker Compose

Файл `compose.yaml` используется для запуска приложения и базы данных.

Он описывает два сервиса:

### `postgres`

Контейнер PostgreSQL:

- использует образ `postgres:16-alpine`;
- получает параметры БД через переменные окружения:
  - `POSTGRES_DB`;
  - `POSTGRES_USER`;
  - `POSTGRES_PASSWORD`;
- содержит `healthcheck`, чтобы приложение запускалось только после готовности базы.

### `app`

Контейнер приложения:

- собирается из текущего каталога;
- получает переменную `DATABASE_URL`;
- получает переменную `PORT`;
- публикует порт наружу.

Связь между контейнерами осуществляется по имени сервиса `postgres`.

То есть внутри Docker-сети приложение обращается к базе не по `127.0.0.1`, а по имени контейнера.

## 15. Переменные окружения

Для приложения и контейнеров важны следующие переменные:

- `DATABASE_URL` — строка подключения приложения к PostgreSQL;
- `POSTGRES_DB` — имя базы данных;
- `POSTGRES_USER` — пользователь PostgreSQL;
- `POSTGRES_PASSWORD` — пароль PostgreSQL;
- `APP_PORT` — порт приложения снаружи контейнера;
- `PORT` — порт, который слушает приложение внутри контейнера.

Это делает решение гибким и пригодным для деплоя на сервер.

## 16. CI/CD workflow

В файле `.github/workflows/docker.yml` описан GitHub Actions workflow.

Он состоит из четырех этапов.

### 16.1. `build`

На этапе сборки:

- репозиторий клонируется на GitHub Runner;
- Docker-образ приложения собирается из каталога `docker`;
- образ сохраняется как artifact.

### 16.2. `test`

На этапе тестирования:

- artifact с образом загружается;
- образ импортируется в Docker;
- через Docker Compose запускается PostgreSQL;
- тесты выполняются внутри уже собранного образа.

Это важный момент: тестирование проходит не просто в Python-окружении раннера, а именно в контейнере приложения. Это соответствует требованию задания.

### 16.3. `deploy`

На этапе деплоя:

- каталог `docker` копируется на сервер по SSH;
- по SSH запускается `docker compose up -d --build --remove-orphans`;
- контейнеры приложения и PostgreSQL поднимаются на сервере.

Деплой выполняется только при `push` в ветку `master`.

### 16.4. `verify`

На этапе проверки:

- выполняется HTTP-запрос к публичному URL вида
  `http://212.192.134.135:60080/<username>/users/`;
- если сервер отвечает без ошибки, деплой считается успешным.

## 17. Секреты GitHub

Для работы деплоя используются GitHub Secrets:

- `DEPLOY_SSH_HOST`;
- `DEPLOY_SSH_USER`;
- `DEPLOY_SSH_KEY`;
- `DEPLOY_PATH`;
- `DEPLOY_USERNAME`;
- `POSTGRES_DB`;
- `POSTGRES_USER`;
- `POSTGRES_PASSWORD`;
- `APP_PORT`.

Они нужны для безопасного хранения настроек сервера и базы данных.

## 20. Ограничения решения

У текущего решения есть и ограничения:

- таблицы создаются через `create_all`, а не через полноценные миграции;
- модель данных минимальна и содержит только имя пользователя;
- отсутствует расширенная валидация входных данных;
- нет разделения на production/dev конфигурации;
- логирование настроено минимально.

Для учебного проекта этого достаточно, но для промышленной эксплуатации обычно добавляют миграции, более строгую валидацию и расширенную конфигурацию.
